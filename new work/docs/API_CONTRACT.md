# AURIX API Contract: Scan Results JSON Schema
**Version:** 1.0
**Producers:** Divyansh (LangGraph AI Engine)
**Consumers:** Bhavya (FastAPI Backend), Bhumika (React Web Dashboard)

## Overview
This document defines the strict JSON payload that the AI Verification Engine will output upon completing a repository scan. 
Bhavya should use this schema to design the PostgreSQL tables. Bhumika should use this to create mock data and build the React interfaces.

---

## The Payload Structure
The engine will send a single JSON object via a Webhook POST request containing the top-level metadata and an array of findings.

### Example JSON Payload
```json
{
  "scan_id": "b05c8ef4-e03b-4186-934f-bd6e5a1a4f75",
  "url": "https://github.com/stamparm/DSVW",
  "timestamp": "2026-07-18T20:57:01.569Z",
  "summary": {
    "total_findings": 13,
    "neutralized_count": 3,
    "scan_engine": "Project AURIX LangGraph v1"
  },
  "findings": [
    // ... Array of Finding Objects
  ]
}
```

---

## The `Finding` Object Schema
Each object inside the `findings` array strictly adheres to the following structure.

### 1. Base Properties (Always Present)
These fields are generated during Phase 1 (The initial static scan by Opengrep/Trivy).
*   `id` **(String/UUID)**: Unique identifier for the finding.
*   `rule_id` **(String)**: The internal rule that triggered (e.g., `python.lang.security.deserialization`).
*   `tool` **(String)**: The scanner that found it (e.g., `opengrep`, `trivy`, `hadolint`).
*   `category` **(String)**: Vulnerability category (`sast`, `infra`, `secret`).
*   `title` **(String)**: Short human-readable title (e.g., `avoid-pickle`).
*   `description` **(String)**: Full explanation of the vulnerability.
*   `severity` **(String)**: Enum `['LOW', 'MEDIUM', 'HIGH', 'CRITICAL']`.
*   `cvss` **(Float)**: 0.0 to 10.0 score.
*   `file` **(String)**: Relative path to the vulnerable file (e.g., `/src/dsvw.py`).
*   `line` **(Integer)**: The line number where the issue exists.
*   `evidence` **(String)**: The raw snippet of code that triggered the alert.
*   `fix` **(String)**: Brief static recommendation.
*   `references` **(Array of Strings)**: URLs to OWASP or documentation.

### 2. Verification Properties (Present if processed by AI)
These fields are appended during Phase 2 (LangGraph Triage and Execution).
*   `verified` **(Boolean)**: 
    *   `true` = The AI confirmed it is a real threat.
    *   `false` = The AI discarded it as a false positive or low-confidence noise.

### 3. Agentic Artifacts (Optional - Only present if `verified: true`)
These fields are the high-value outputs generated by the Red and Blue Agents.
*   `wargame_status` **(String)**: 
    *   `"PoC Failed"` = AI tried to exploit it in Docker but failed.
    *   `"Exploit Confirmed"` = Red Agent successfully exploited it, but Blue Agent couldn't patch it.
    *   `"Neutralized"` = Perfect success. Exploit confirmed AND patch mathematically proven to work.
*   `ai_reasoning` **(String)**: The LLM's explanation of why this bug is real and how it works.
*   `poc_script` **(String)**: Raw Python/Bash code of the exploit script used in the Docker Sandbox.
*   `patch_code` **(String)**: Raw Python code representing the Blue Agent's patch file (intended to be merged).
